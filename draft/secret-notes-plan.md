# obsidian-secret-notes 需求与实现说明

## 1. 目标

这个插件用于在 Obsidian 中管理 `secret` 代码块，让敏感内容在编辑体验上可按需查看，但在文件持久化时保持加密状态。

目标效果有三点：

1. 用户可以写入明文 `secret` 代码块，并在首次处理时将其转换为加密 JSON。
2. 已加密的 `secret` 代码块在阅读视图中不会直接显示密文，而是显示一个带“解密”按钮的占位区域。
3. 用户解密后可以查看明文，但在文件保存前，内容必须再次以原密码加密，避免明文落盘。

## 2. 功能定义

### 2.1 输入形态

用户会先写一个代码块：

```markdown
```secret
这里是敏感内容
```
```

插件不会在用户输入时立刻打断编辑，而是在以下时机处理该代码块：

1. 切换到预览模式时。
2. 保存文件时。

如果代码块内容仍然是明文，插件会弹窗请求密码，并使用该密码对代码块内部文本进行加密。

### 2.2 存储形态

加密完成后，原来的明文内容会被替换为 JSON，结构参考 [draft/secret.json](draft/secret.json)：

```json
{
  "v": 1,
  "title": "string",
  "hint": "string",
  "encrypted": "...",
  "date": "2026-06-30 12:34:56"
}
```

字段理解如下：

1. `v`：密文格式版本号，便于未来升级加密方案。
2. `title`：可选标题，在首次加密时由用户选择性填写，用来描述该块保存了什么内容。
3. `hint`：密码提示，在首次加密时由用户选择性填写，不在第一次解密尝试前显示。
4. `encrypted`：密文主体，建议存放 AES 加密后的字符串。
5. `date`：加密时间，便于调试与后续兼容处理。

最终写入 Markdown 文件的不是明文，而是这段 JSON。

### 2.3 渲染形态

当 Obsidian 渲染到一个 `secret` 代码块，且其内容已经是上述 JSON 结构时，不应直接显示 JSON 文本，而应渲染为自定义区域，例如：

1. 显示“已加密”字样。
2. 显示“解密”按钮。
3. 如果 `title` 存在，应优先显示标题，帮助用户识别内容。
4. 可以附带显示加密时间等轻量元信息。

### 2.4 解密流程

当用户点击“解密”按钮时：

1. 插件弹窗要求输入密码。
2. 使用该密码尝试解密 `encrypted` 字段。
3. 如果密码正确，则在当前视图或当前编辑上下文中展示解密后的明文。
4. 如果密码错误，则提示用户解密失败，不泄露任何原文。
5. 第一次输入错误后，开始显示该块的 `hint` 字段。

密码错误次数和等待策略定义如下：

1. 前 4 次输错只提示失败，不增加等待时间。
2. 从第 5 次开始增加等待时间。
3. 之后每累计 3 次额外失败，就提升下一次可尝试前的等待时间等级。
4. 等待时间按二次函数增长，实现上可以采用“等级平方后换算为秒数”的方式。
5. 一旦解密成功，该块的失败计数和等待状态立即清零。

### 2.5 保存前重加密

这是需求中的关键约束。

如果用户已经成功解密，并且当前会话中 `secret` 块处于明文可见或可编辑状态，那么在 Obsidian 将文件写回磁盘之前，插件必须使用此前输入过的密码重新加密该块，并把内容恢复成 JSON 结构。

这意味着：

1. 明文可以在内存中短暂存在。
2. 明文不应直接写入最终保存的 Markdown 文件。
3. 插件需要在当前文件范围内暂存该 `secret` 块对应的密码或等价的重加密能力。
4. 密码缓存粒度是“一个块一个密码”，不能把多个块默认视为共用密码。

## 3. 推荐的用户流程

### 3.1 首次加密

1. 用户输入明文 `secret` 代码块。
2. 用户切换到预览模式，或触发保存。
3. 插件检测到该代码块内容不是合法密文 JSON。
4. 插件弹出表单，请求输入密码，并可选输入 `title` 和 `hint`。
5. 用户确认后，插件立即将明文替换为加密 JSON。
6. 预览模式下显示“已加密”占位区域。

### 3.2 查看明文

1. 用户在阅读视图看到“已加密”区域。
2. 点击“解密”按钮。
3. 输入密码。
4. 如果失败，提示错误；从第一次失败后开始显示 `hint`。
5. 如果成功，插件校验通过并显示明文。
6. 解密成功后，当前块对应密码缓存在当前文件会话中，供再次加密使用。

### 3.3 编辑已解密内容

1. 解密成功后，明文内容允许继续编辑。
2. 用户对明文的修改应被视为该块的新原文。
3. 该块在当前文件关闭前都可以保持“已解密且可编辑”的会话状态。

### 3.4 再次保存

1. 用户可能在当前会话中已经看过或编辑过明文。
2. 文件保存前，插件扫描当前文档中的相关 `secret` 块。
3. 对处于明文状态的块重新执行加密。
4. 重加密时优先复用该块当前文件会话中缓存的密码。
5. 最终落盘内容仍是密文 JSON。

## 4. 实现拆分建议

### 4.1 数据层

需要定义一个统一的数据结构，例如：

```ts
type SecretPayload = {
  v: number;
  title?: string;
  hint?: string;
  encrypted: string;
  date: string;
};
```

并提供两个基础能力：

1. 判断代码块内容是明文还是合法密文 JSON。
2. 在明文和 `SecretPayload` 之间做转换。
3. 校验 `title` 和 `hint` 等附加字段是否合法。

### 4.2 加密层

需要封装：

1. `encrypt(plainText, password, meta) -> SecretPayload`
2. `decrypt(payload, password) -> plainText`
3. `shouldThrottle(failureCount, lastFailureAt) -> waitState`

这里建议从一开始就固定版本号与算法参数，避免后面难以兼容。

### 4.3 UI 层

至少需要两个交互组件：

1. 首次加密弹窗，用于输入密码，并可选填写 `title` 和 `hint`。
2. 解密弹窗，用于输入密码，并在第一次失败后显示 `hint`。
3. 已加密块的渲染组件。

渲染组件负责显示状态、触发解密，并在成功后切换显示内容。

### 4.4 文档生命周期层

需要接入 Obsidian 的文档事件或保存前钩子，处理两个动作：

1. 在切换预览模式时，首次发现明文 `secret` 块则触发加密。
2. 在文档保存前，对当前处于明文状态的块重新加密。
3. 在当前文件作用域维护“块标识 -> 密码/状态”的运行时缓存。

这一层决定了插件是否真正满足“明文不落盘”的核心要求。

## 5. 已确定的实现约束

1. 自动加密触发时机固定为“切换预览”和“保存文件”。
2. 解密后的明文允许继续编辑。
3. 密码缓存范围固定为“当前文件内有效”。
4. 密码缓存粒度固定为“一个块一个密码”。
5. 密码错误允许无限次重试，但需要按失败次数施加递增等待时间。

## 6. 建议的最小可用版本

第一版可以先只做最小闭环：

1. 支持识别 `secret` 代码块。
2. 支持首次输入密码，并附带可选的 `title` 和 `hint`，然后把明文替换成 JSON。
3. 支持阅读视图中的“已加密 + 解密按钮”渲染。
4. 支持输入密码后显示明文。
5. 支持密码错误计数、提示显示和基础等待限制。
6. 支持保存前重加密。

先不做复杂能力，例如全局密码管理、多算法切换、批量重加密、历史版本兼容等。

## 7. 一句话总结

这个插件的本质不是“展示密文”，而是为 Obsidian 提供一种 `secret` 代码块的安全生命周期管理：明文输入后在预览或保存时加密，阅读时可按块解密与编辑，最终保存时强制回到带元信息的密文 JSON。
